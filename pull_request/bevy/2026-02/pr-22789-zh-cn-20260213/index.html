描述翻译

目标

  • 将生命周期“Replace”钩子/事件重命名为“Discard”，以反映其实际语义（在组件被覆写、移除和实体反生成时触发），并为未来真正的替换事件腾出“Replace”名称。解决 #20729。

解决方案

  • 在钩子、观察者、事件、派生宏、文档、示例和测试中，将 Replace/on_replace 重命名为 Discard/on_discard。
  • 更新了 UI 编译失败测试的预期输出，并为 Discard 添加了 OnDiscard 文档别名以增强可发现性。

测试

  • cargo run -p ci -- compile-fail
  • cargo test -p bevy_ecs --lib --tests --features bevy_ecs/track_location （本地因两个测试缺少 bevy_ecs/debug 特性而失败，这两个测试断言了完整的系统名称）

本PR的故事

此PR源于一个命名问题。在 Bevy 的 ECS 系统中，有一个名为 Replace 的生命周期事件，其关联的组件钩子名为 on_replace。根据其设计，这个事件不仅在新值替换旧值时触发（即“替换”），还会在组件被移除（remove）或实体被反生成（despawn）时触发。这使得“Replace”这个名字具有误导性，因为它暗示的场景（一个值替换另一个）只是其触发条件的子集。

问题 #20729 指出了这一点，并建议进行重命名。不准确的命名会给开发者带来困惑，增加学习成本，并可能导致对 API 行为的错误假设。此外，为“替换”这个更具体的语义保留 Replace 这个名字，为未来的 API 改进留出了空间。

因此，该PR的目标非常直接：用一个能准确反映其“丢弃”旧值这一核心语义的名字来替换 Replace 和 on_replace。Discard 一词准确地描述了无论后续是否插入新值，旧值都会被丢弃（覆写、移除或随实体销毁）的场景。这种重命名属于API的清理和语义精确化，不改变任何功能逻辑，是一个纯粹的重构。

解决此问题的方法是对整个代码库进行系统性的搜索和替换。这远不止是简单的文本替换，因为它需要修改多个不同层级的代码：

  1. 核心数据类型：事件结构体（Replace -> Discard）和组件钩子字段名（on_replace -> on_discard）。
  2. 常量和枚举：代表该事件的常量（REPLACE -> DISCARD）以及在各种标志位（如 ArchetypeFlags）中的枚举项。
  3. 过程宏：#[derive(Component)] 宏支持的 #[component(on_replace = ...)] 属性需要改为 #[component(on_discard = ...)]。宏内部的解析、验证和代码生成逻辑也需要更新。
  4. 内部方法：一系列用于触发这些钩子和观察者的内部方法名需要更新，例如 trigger_on_replace -> trigger_on_discard，has_replace_observer -> has_discard_observer。
  5. 文档和注释：所有相关的文档字符串、示例代码和代码注释都需要更新，以反映新的术语。
  6. 测试代码：包括单元测试、集成测试以及 UI 编译失败测试的预期错误信息。

让我们深入几个关键文件，看看变化是如何具体实现的。

在 crates/bevy_ecs/src/lifecycle.rs 中，我们看到了核心定义的变化：

// 之前：
/// 当组件从实体上被移除时触发的事件，无论其后是否被新值替换。
#[doc(alias = "OnReplace")]
pub struct Replace {
    pub entity: Entity,
}
pub const REPLACE: EventKey = EventKey(ComponentId::new(crate::component::REPLACE));

// 之后：
/// 当组件从实体上被移除时触发的事件，无论其后是否被新值替换。
#[doc(alias = "OnDiscard")]
#[doc(alias = "OnReplace")] // 保留旧别名以兼容旧文档链接？
#[doc(alias = "Replace")]
pub struct Discard {
    pub entity: Entity,
}
pub const DISCARD: EventKey = EventKey(ComponentId::new(crate::component::DISCARD));

注意这里添加了 #[doc(alias = "OnReplace")] 和 #[doc(alias = "Replace")]。这是一个重要的用户体验优化。它意味着开发者在文档中搜索 Replace 或 OnReplace 时，仍然能被引导到正确的 Discard 类型，确保了在重命名后的可发现性。

ComponentHooks 结构体的字段也相应更改：

// 之前：
pub(crate) struct ComponentHooks {
    pub(crate) on_add: Option<ComponentHook>,
    pub(crate) on_insert: Option<ComponentHook>,
    pub(crate) on_replace: Option<ComponentHook>, // 被替换
    pub(crate) on_remove: Option<ComponentHook>,
    pub(crate) on_despawn: Option<ComponentHook>,
}

// 之后：
pub(crate) struct ComponentHooks {
    pub(crate) on_add: Option<ComponentHook>,
    pub(crate) on_insert: Option<ComponentHook>,
    pub(crate) on_discard: Option<ComponentHook>, // 新字段名
    pub(crate) on_remove: Option<ComponentHook>,
    pub(crate) on_despawn: Option<ComponentHook>,
}

同时，所有相关的方法名也从 on_replace / try_on_replace 更新为 on_discard / try_on_discard。

在派生宏 (crates/bevy_ecs/macros/src/component.rs) 中的改动展示了如何保持编译时检查。解析属性列表的逻辑需要更新：

// 之前：
} else if nested.path.is_ident(ON_REPLACE) {
    attrs.on_replace = Some(HookAttributeKind::parse(nested.input, || {
        parse_quote! { Self::on_replace }
    })?);
    Ok(())

// 之后：
} else if nested.path.is_ident(ON_DISCARD) {
    attrs.on_discard = Some(HookAttributeKind::parse(nested.input, || {
        parse_quote! { Self::on_discard }
    })?);
    Ok(())

这里 ON_REPLACE 常量被改为 ON_DISCARD。更重要的是，当宏检测到用户试图在关系组件上定义自定义的 on_discard 钩子时，它现在会抛出一个语义更准确的错误信息：

// 错误信息更新：
// 之前： “Custom on_replace hooks are not supported as RelationshipTarget already defines an on_replace hook”
// 之后： “Custom on_discard hooks are not supported as RelationshipTarget already defines an on_discard hook”

在 crates/bevy_ecs/src/observer/centralized_storage.rs 中，观察者缓存的结构体字段也进行了重命名，这反映了从 replace 到 discard 的命名贯穿了整个观察者系统：

// 之前：
pub struct Observers {
    add: CachedObservers,
    insert: CachedObservers,
    replace: CachedObservers, // 被替换
    remove: CachedObservers,
    despawn: CachedObservers,
}

// 之后：
pub struct Observers {
    add: CachedObservers,
    insert: CachedObservers,
    discard: CachedObservers, // 新字段名
    remove: CachedObservers,
    despawn: CachedObservers,
}

类似地，在 crates/bevy_ecs/src/archetype.rs 中，用于快速检查的原型标记位也进行了更新：

// ArchetypeFlags 位标志更新：
const ON_REPLACE_HOOK = (1 << 2);    -> const ON_DISCARD_HOOK = (1 << 2);
const ON_REPLACE_OBSERVER = (1 << 7); -> const ON_DISCARD_OBSERVER = (1 << 7);

这些标记位被诸如 has_discard_hook 和 has_discard_observer 等方法使用，这些方法在组件插入和移除路径中被调用，以决定是否需要运行相应的钩子或通知观察者。将这些内部方法名一并更新，保持了代码的一致性。

此次重命名的影响是全栈的，从最底层的常量到最高级的用户API。它解决了核心的术语混淆问题，并使 Replace 这个名字可用于未来可能引入的、仅在值被实际替换（而非移除）时触发的更精确的事件。对于现有项目，这无疑是一个破坏性变更（breaking change），但PR描述中附带的迁移指南（release-content/migration-guides/lifecycle-events.md）提供了清晰的升级路径：将所有 Replace 引用改为 Discard，将所有 on_replace 引用改为 on_discard。

从工程角度看，这次重构执行得相当彻底。它通过了现有的测试套件（包括编译失败测试），并且作者考虑到了开发者体验，通过 #[doc(alias)] 来缓解搜索时的困扰。这是一个典型的语义清理工作，虽然改动范围广泛，但每个修改都直接、一致地服务于同一个明确的目标。
