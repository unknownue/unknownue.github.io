    Objective

    • Fixes #14949
    • Currently an Option<Gizmos> causes a panic when the config group backing Gizmos has not been initialized. The desired behavior is to have it be None instead of course.

    Solution

    Gizmos is a non #[derive]’d SystemParam that requires its

    • GizmoConfigGroup to have an entry in GizmoConfigStore
    • Various GizmoStorages to be inserted as resources.

    In order to support a non-panicking Option<Gizmos> SystemParam in a non-intrusive way for users when the Gizmo Config Group has not been set up intentionally, this PR:

    • Adds ergonomic methods to get a group’s config fallibly to the GizmoConfigStore
    • Changes the validate_param method of Gizmos to return an error in the event that it could not find its own config in the GizmoConfigStore
    • Changes the apply method of GizmoBuffer to only apply changes if the GizmoStorage for the config group exists.

    Testing

    I modified the 2d gizmos example (cargo run --example 2d_gizmos) by:

    • commenting out the init of the custom gizmo group // .init_gizmo_group::<MyRoundGizmos>()
    • Using options (i.e. Option<Gizmos> and Option<Gizmos<MyRoundGizmos>>) for the two Gizmos system parameters in draw_example_collection and requiring my_gizmos_option.is_some() before using an unwrapped my_gizmos to draw things.
    • Using the new GizmoConfigStore method try_config_mut when trying to get the config for MyRoundGizmos, and only proceeding to change its settings if my_config_option.is_some()

    There are no crashes. Default Gizmos continue to be drawn, and MyRoundGizmos are not drawn. Changing the settings still works for default gizmos, and has no effect if they were to be applied to MyRoundGizmos. There seems to be no deterioration in performance for the example.

    The Story of This Pull Request

    The issue began with a specific scenario in Bevy’s ECS (Entity Component System) where using Option<Gizmos> as a system parameter would cause a panic. This happened when the corresponding GizmoConfigGroup hadn’t been initialized via app.init_gizmo_group<T>(). The expected behavior was that Option<Gizmos> should simply be None in such cases, not crash the application.

    The root cause was in how the Gizmos system parameter validated itself. Gizmos is a custom SystemParam (not derived via #[derive(SystemParam)]) that requires two conditions to be met: its GizmoConfigGroup must have an entry in the GizmoConfigStore, and the corresponding GizmoStorage resources must be present. When using Option<Gizmos>, Bevy’s system parameter validation still runs, and the original implementation would panic when it couldn’t find the required configuration.

    The solution involved three coordinated changes that work together to provide graceful failure handling. First, the GizmoConfigStore was extended with fallible methods get_config() and get_config_mut() that return Option types instead of panicking. These complement the existing panicking methods config() and config_mut(), giving users the choice between guaranteed existence (with panic on missing config) or optional existence (with None return).

    // New fallible method in GizmoConfigStore
pub fn get_config<T: GizmoConfigGroup>(&self) -> Option<(&GizmoConfig, &T)> {
    let (config, ext) = self.get_config_dyn(&TypeId::of::<T>())?;
    let ext = ext.as_any().downcast_ref().unwrap();
    Some((config, ext))
}

    The second change was in the Gizmos system parameter’s validate_param() method. Previously, it delegated validation entirely to GizmosState. The updated version adds an additional check after the delegation: it attempts to retrieve the configuration for the specific Config type, and if it doesn’t exist, returns a SystemParamValidationError instead of allowing the system to continue in an invalid state.

    fn validate_param(
    &mut self,
    system_meta: &SystemMeta,
    world: UnsafeWorldCell,
) -> Result<(), SystemParamValidationError> {
    // SAFETY: Delegated to existing `SystemParam` implementation.
    unsafe {
        GizmosState::<Config, Clear>::validate_param(&mut state.state, system_meta, world)?;
    }

    // SAFETY: Delegated to existing `SystemParam` implementation.
    let (_, f1) = unsafe {
        GizmosState::<Config, Clear>::get_param(
            &mut state.state,
            system_meta,
            world,
            world.change_tick(),
        )
    };
    // This if-block is to accommodate an Option<Gizmos> SystemParam.
    // The user may decide not to initialize a gizmo group, so its config will not exist.
    if f1.get_config::<Config>().is_none() {
        Err(SystemParamValidationError::invalid::<Self>(
            format!("Requested config {} does not exist in `GizmoConfigStore`! Did you forget to add it using `app.init_gizmo_group<T>()`?", 
            Config::type_path())))
    } else {
        Ok(())
    }
}

    This validation error is what allows Option<Gizmos> to become None when the configuration is missing, as Bevy’s system parameter handling interprets validation failures for optional parameters as “parameter should be None”.

    The third change addresses a potential memory leak in the GizmoBuffer’s queue() method. Previously, this method would always try to append buffer data to the corresponding GizmoStorage resource. If the GizmoStorage didn’t exist (because the config group wasn’t initialized), the buffer would keep accumulating data indefinitely. The fix checks for the existence of the resource first, and only appends data if it exists. If not, it clears the buffers to prevent unbounded growth.

    fn queue(&mut self, _system_meta: &SystemMeta, mut world: DeferredWorld) {
    if let Some(mut storage) = world.get_resource_mut::<GizmoStorage<Config, Clear>>() {
        storage.list_positions.append(&mut self.list_positions);
        storage.list_colors.append(&mut self.list_colors);
        storage.strip_positions.append(&mut self.strip_positions);
        storage.strip_colors.append(&mut self.strip_colors);
    } else {
        // Prevent the buffer from growing indefinitely if GizmoStorage
        // for the config group has not been initialized
        self.list_positions.clear();
        self.list_colors.clear();
        self.strip_positions.clear();
        self.strip_colors.clear();
    }
}

    These three changes work together to create a cohesive solution: the fallible config access methods provide the building blocks, the validation change makes Option<Gizmos> behave correctly, and the buffer change prevents resource leaks. The approach is non-intrusive for users - existing code continues to work, and the new behavior only affects scenarios where Option<Gizmos> is used without proper initialization.

    The testing methodology was practical and thorough. By modifying the existing 2d gizmos example to comment out the initialization of a custom gizmo group and using Option<Gizmos> parameters, the author verified that the system no longer panics. The test also confirmed that default gizmos continue to work normally, uninitialized custom gizmos remain undrawn, and there’s no performance degradation.

    Key Files Changed

    crates/bevy_gizmos/src/config.rs (+40/-4)

    This file was modified to add fallible methods for accessing gizmo configurations. The existing config() and config_mut() methods were refactored to use the new get_config() and get_config_mut() methods internally, maintaining backward compatibility while adding the new functionality.

    Key changes:

    // Before (simplified):
pub fn config<T: GizmoConfigGroup>(&self) -> (&GizmoConfig, &T) {
    let Some((config, ext)) = self.get_config_dyn(&TypeId::of::<T>()) else {
        panic!("Requested config {} does not exist...", T::type_path());
    };
    let ext = ext.as_any().downcast_ref().unwrap();
    (config, ext)
}

// After:
pub fn config<T: GizmoConfigGroup>(&self) -> (&GizmoConfig, &T) {
    let Some(configs) = self.get_config() else {
        panic!("Requested config {} does not exist...", T::type_path());
    };
    configs
}

pub fn get_config<T: GizmoConfigGroup>(&self) -> Option<(&GizmoConfig, &T)> {
    let (config, ext) = self.get_config_dyn(&TypeId::of::<T>())?;
    let ext = ext.as_any().downcast_ref().unwrap();
    Some((config, ext))
}

    These changes provide a clean API where users can choose between guaranteed-existence (with panic) or fallible access patterns, which is essential for the Option<Gizmos> use case.
