描述翻译

目标 SLASHED_ZERO 这个 FontFeatureTag 常量被设置为 “ordn”，但正确的标签应该是 “zero”。

https://en.wikipedia.org/wiki/List_of_typographic_features

解决方案 将其设置为 “zero”。

本次 PR 的故事

这个 PR 虽然改动很小，但处理了一个典型的软件开发问题：一个常量被错误地赋值。它涉及 Bevy 游戏引擎的文本渲染系统，具体来说是 FontFeatureTag 结构体中定义的一个常量。

问题与背景 在 Bevy 的文本渲染模块中，开发者使用 FontFeatureTag 来表示 OpenType 字体特性标签。这些标签是四个字符的标识符，用于启用或禁用特定的字体渲染特性，例如连字、替代数字形式等。其中有一个特性是 “SLASHED_ZERO”，它用于显示带斜杠的零（“0̸” 或类似效果），以便在视觉上与字母 “O” 区分开，这在某些字体和上下文中（如编程、技术文档）非常有用。

然而，问题在于代码中 SLASHED_ZERO 常量被错误地赋值为 “ordn”。根据 OpenType 特性标签列表，“ordn” 标签对应的是 “Ordinals”（序数词）特性，它会将字母上标（如 “1st” 中的 “st”）格式化为序数词形式。这显然不是 “SLASHED_ZERO” 想要实现的功能。

这个错误的根源很可能是一个笔误或复制粘贴错误。在代码中，SLASHED_ZERO 常量的定义紧挨着 ORDINALS 常量的定义，两个常量都使用了 FontFeatureTag::new(b"ordn") 来初始化。这表明开发者可能在定义 SLASHED_ZERO 时错误地复制了上一行的值，而没有将其改为正确的标签 “zero”。

解决方案 解决方案非常直接：将 SLASHED_ZERO 常量的值从 “ordn” 更正为 “zero”。这个改动遵循了 OpenType 规范，确保了代码的意图（启用带斜杠的零）与实际行为（使用正确的字体特性标签）一致。

实现细节 具体的实现只涉及一个文件的单行修改。下面是关键代码段：

// 在 crates/bevy_text/src/text.rs 中，第 760 行附近
// 之前：
pub const SLASHED_ZERO: FontFeatureTag = FontFeatureTag::new(b"ordn");

// 之后：
pub const SLASHED_ZERO: FontFeatureTag = FontFeatureTag::new(b"zero");

这个修改纠正了常量值，使得当开发者使用 TextStyle 并启用 SLASHED_ZERO 特性时，渲染器会向底层文本渲染栈（如 glyphon 或 cosmic-text）传递正确的 OpenType 特性标签 “zero”，从而触发正确的字体渲染行为。

技术洞察

  1. 常量定义的正确性：这个错误凸显了在定义常量时，特别是当多个相似常量相邻时，仔细检查每个值的重要性。尽管现代编译器无法捕获这种语义错误（因为 “ordn” 和 “zero” 都是有效的字节字面量），但代码审查和良好的测试覆盖可以帮助预防这类问题。
  2. 字体特性标签：OpenType 字体特性标签是四个字符的 ASCII 代码，通常用小写字母表示。这些标签是字体渲染中一个相对底层的概念，但 Bevy 通过 FontFeatureTag 和相关的文本样式 API 将其暴露给开发者，提供了对文本外观的精细控制。
  3. 向后兼容性：这个修改是向后兼容的，因为它只是纠正了一个错误。任何依赖错误 “ordn” 值的代码本来就没有得到预期的 “SLASHED_ZERO” 行为，而是得到了 “ORDINALS” 行为。修正后，这些代码将开始按预期工作。不过，如果某个应用恰好依赖这个错误行为（即，当设置 SLASHED_ZERO 时意外获得了序数词效果），那么修正后其行为会改变。但这种可能性极低，因为这两个特性在语义上完全不同。

影响 这个修正虽然微小，但很重要：

  • 功能正确性：确保了 SLASHED_ZERO 特性按预期工作。
  • 代码可维护性：消除了一个令人困惑的错误，使得其他开发者或未来的维护者不会被这个不一致的常量所误导。
  • 文档一致性：与 OpenType 规范 保持了一致，后者明确将 “zero” 标签定义为 “Slashed Zero”。

总的来说，这是一个典型的错误修正：识别并修复一个简单的常量赋值错误，以确保代码的功能与其设计意图一致。这类修正虽然看起来微不足道，但却是保持代码库健康和可靠性的重要组成部分。
