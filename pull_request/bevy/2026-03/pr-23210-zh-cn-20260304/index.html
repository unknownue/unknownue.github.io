Description Translation

目标

bevy_platform 可选择性地使用 wasm-bindgen-futures/std 特性，但该特性在 0.4.46 版本之前不存在。当一个项目拥有长期存在的锁文件（lockfile）或有意使用最小版本约束时，添加 Bevy 依赖可能导致构建失败。

解决方案

将 bevy_platform → wasm-bindgen-futures 的最低版本要求提升至 0.4.46，而不是 0.4。

测试

除了 CI 测试外未进行额外测试，但此更改很难破坏任何功能。

（针对此类 PR 修复的缺陷的测试在 #9593 中。）

The Story of This Pull Request

这个 PR 解决了一个由于对可选依赖特性的版本约束不精确而导致的潜在构建失败问题。bevy_platform crate 是 Bevy 引擎中用于抽象不同平台（如 WASM、Android、iOS）特定代码的模块。为了支持 WebAssembly 环境下的异步操作，它条件性地依赖 wasm-bindgen-futures crate，并启用了其可选的 std 特性。

问题的核心在于 Cargo 的版本解析机制和可选特性（feature）的可用性。在 Cargo.toml 中，依赖版本 "0.4" 是一个宽容的约束，它允许 Cargo 解析到该依赖的任何 0.4.x 版本，甚至是最早的 0.4.0。然而，bevy_platform 在针对 target_arch = "wasm32" 时，其代码逻辑依赖于 wasm-bindgen-futures 的 std 特性。

根据 PR 描述，wasm-bindgen-futures/std 这个特性是在该 crate 的 0.4.46 版本中才被引入的。这意味着，如果一个项目因为锁文件（Cargo.lock）长期未更新，或者因为执行了 cargo update -Z minimal-versions（一种用于验证最低版本兼容性的工作流）而解析到了 wasm-bindgen-futures 的 0.4.0 到 0.4.45 之间的任何版本，那么在尝试构建启用了相关特性的 bevy_platform 时，Cargo 会因为找不到 std 特性而报错。

开发者 kpreid 采取的解决方案直接且有效：将版本约束从模糊的 "0.4" 收紧到明确的 "0.4.46"。这个修改确保了任何被解析到的 wasm-bindgen-futures 版本都必定包含 std 特性，从而消除了构建失败的风险。

从工程角度看，这是一个对向后兼容性（backwards compatibility）的典型维护操作。修改本身是稳健的（robust），因为它只提升了最低兼容版本，而没有改变任何实际调用的 API。正如作者在测试说明中提到的，这个更改“很难破坏任何东西”。同时，作者也指出了更系统性地测试这类“最小版本”问题的挑战，并将其指向了相关的 Issue #9593，这体现了对工程问题上下文的清晰认识。

这个 PR 虽然修改量极小，但它凸显了在管理具有平台特定可选依赖的大型项目时，精确声明版本约束的重要性。它防止了用户在实际开发中可能遇到的一个隐晦的构建错误，提升了开发体验和 crate 的可靠性。

Key Files Changed

  • crates/bevy_platform/Cargo.toml (+1/-1)
  1. 描述：此文件定义了 bevy_platform crate 的元数据和依赖关系。唯一的修改是针对 wasm-bindgen-futures 依赖的版本号字符串，旨在确保构建系统总能获取到包含必需特性的足够新的版本。
  2. 代码变更：
    # File: crates/bevy_platform/Cargo.toml
# Before:
wasm-bindgen-futures = { version = "0.4", default-features = false, optional = true }

# After:
wasm-bindgen-futures = { version = "0.4.46", default-features = false, optional = true }
  3. 与 PR 目标的关联：这个单行修改直接实现了 PR 的目标，即通过设置精确的最低版本号来修复因可选特性在早期版本中不存在而导致的潜在构建失败。
