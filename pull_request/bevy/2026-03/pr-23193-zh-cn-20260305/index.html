该 Pull Request 的来龙去脉

这个 PR 的核心是对 StaticTransformOptimizations 资源进行了一次简化和重构。原来的设计引入了一个动态阈值机制，本意是在场景的动态程度超过某个百分比时自动禁用静态变换优化，以避免跟踪开销超过性能收益。然而，这个机制在实践中带来了一些问题。

问题与背景 StaticTransformOptimizations 原本是一个包含阈值（threshold）和启用状态（enabled）字段的资源。其逻辑是：在每一帧的 mark_dirty_trees 系统中，它会计算场景中动态（变换发生改变）实体的比例。如果这个比例超过了预设的阈值（默认 0.3，即 30%），则临时禁用优化。这样设计的初衷是好的——试图在静态场景和高度动态场景之间自动找到最佳平衡点。

但问题在于，这种动态计算本身就有开销。每一帧都需要统计两个 Query 的结果（变换改变的实体数和总实体数），进行浮点除法和比较。对于大型场景，即使是简单的计数操作也可能带来不可忽视的成本。此外，这种“智能”的、基于运行时而定的行为使得系统的性能表现变得不那么可预测，增加了调试的复杂性。

解决方案与实现 开发者采取的解决方案是彻底移除阈值机制，将 StaticTransformOptimizations 简化为一个简单的 enum。现在它只有两个明确的状态：Enabled 和 Disabled（Enabled 为默认值）。这意味着优化要么完全启用，要么完全禁用，决策权完全交给了开发者，而不是在运行时由引擎自动判断。

这个改动涉及几个关键部分：

  1. 资源定义的重构：StaticTransformOptimizations 从一个结构体变成了一个枚举。移除了 threshold 和 enabled 字段，以及相关的构造函数（from_threshold, disabled, enabled）。现在，判断优化是否启用只需要检查枚举值是否等于 StaticTransformOptimizations::Enabled，这个逻辑被封装在新增的 is_enabled() 方法里。

    // 之前：一个包含复杂状态的结构体
#[derive(Resource, Debug)]
pub struct StaticTransformOptimizations {
    threshold: f32,
    enabled: bool,
}

// 之后：一个简单的枚举，状态明确
#[derive(Resource, Debug, Default, PartialEq, Eq)]
pub enum StaticTransformOptimizations {
    #[default]
    Enabled,
    Disabled,
}

  2. 系统逻辑的简化：mark_dirty_trees 系统是变化的核心。原来需要读取阈值、计算动态实体比例并进行判断的复杂逻辑被完全移除。现在，该系统只需检查 static_optimizations.is_enabled()，如果为假则直接返回，跳过后续所有为静态优化服务的标记工作。

    // 之前：复杂的阈值计算逻辑
let threshold = static_optimizations.threshold.clamp(0.0, 1.0);
match threshold {
    0.0 => static_optimizations.enabled = false,
    1.0 => static_optimizations.enabled = true,
    _ => {
        static_optimizations.enabled = true;
        let n_dyn = changed_transforms.count() as f32;
        let total = transforms.count() as f32;
        if n_dyn / total > threshold {
            static_optimizations.enabled = false;
        }
    }
}
if !static_optimizations.enabled {
    return;
}

// 之后：直接检查是否启用
if !static_optimizations.is_enabled() {
    return;
}

  3. 传播系统的适配：在并行（parallel）和单线程（single_threaded）的变换传播系统中，原来检查 static_optimizations.enabled 的地方，现在都改为调用 static_optimizations.is_enabled()。这保持了行为的一致性，只是判断的依据从内部可变状态变成了不变的枚举值。

技术洞见与影响 这次重构体现了几个重要的软件工程原则：

  • KISS (Keep It Simple, Stupid): 移除了不必要的复杂性。动态阈值是一个“聪明”的特性，但它的收益（自动适应场景）可能不敌其成本（运行时开销和认知负担）。将其替换为明确的手动控制，使系统行为更可预测、更易于理解。
  • 关注点分离: 将“是否应该启用优化”这个策略决策从引擎核心逻辑中剥离出来。引擎现在只负责高效地执行“启用”或“禁用”状态下的逻辑。策略决策（如何选择状态）可以完全由上层应用开发者根据具体场景决定。
  • 性能优化: 移除了每帧的计数和浮点运算，为性能敏感的场景（尤其是实体数量庞大的情况）带来了直接的、确定性的性能提升。count() 操作虽然看起来简单，但在大型 Query 上执行仍然有成本。

迁移与影响 这个改动是一次破坏性变更（breaking change）。之前使用 StaticTransformOptimizations::from_threshold()、::disabled() 或 ::enabled() 的代码需要更新。用户现在需要明确选择 StaticTransformOptimizations::Enabled 或 StaticTransformOptimizations::Disabled。PR 中更新了 bevymark 和 many_foxes 这两个压力测试例子，展示了如何将 ::disabled() 替换为 ::Disabled。

对于确实需要根据场景动态程度来切换优化状态的用户，迁移指南给出了建议：可以在自己的系统中统计实体数量，然后动态地插入或修改 StaticTransformOptimizations 资源。但指南也明智地指出，这种检查本身可能较慢，因此不建议每帧都进行。这实际上是将性能权衡的掌控权和责任明确地交给了开发者。

总而言之，这个 PR 通过简化 StaticTransformOptimizations 的设计，提高了代码的清晰度和运行时的可预测性，并可能带来轻微的性能改善。它将一个隐含的、自动的优化策略转变为一个明确的、由开发者控制的开关，这符合 Bevy 引擎追求明确性和性能的设计哲学。

关键文件变更

crates/bevy_transform/src/systems.rs (+16/-64)

这是本次重构的核心文件，StaticTransformOptimizations 的定义和相关系统逻辑都在这里。

  1. 资源定义重构：将结构体改为枚举，移除了阈值逻辑。

    // 之前：
#[derive(Resource, Debug)]
pub struct StaticTransformOptimizations {
    threshold: f32,
    enabled: bool,
}
impl StaticTransformOptimizations {
    pub fn from_threshold(threshold: f32) -> Self { ... }
    pub fn disabled() -> Self { ... }
    pub fn enabled() -> Self { ... }
}
impl Default for StaticTransformOptimizations {
    fn default() -> Self {
        Self { threshold: 0.3, enabled: true }
    }
}

// 之后：
#[derive(Resource, Debug, Default, PartialEq, Eq)]
pub enum StaticTransformOptimizations {
    #[default]
    Enabled,
    Disabled,
}
impl StaticTransformOptimizations {
    #[inline]
    pub fn is_enabled(&self) -> bool {
        *self == StaticTransformOptimizations::Enabled
    }
}

  2. mark_dirty_trees 系统简化：移除阈值计算，直接使用 is_enabled()。

    // 之前：参数为 ResMut<StaticTransformOptimizations>，内部会修改 enabled 状态。
pub fn mark_dirty_trees(
    ...,
    mut static_optimizations: ResMut<StaticTransformOptimizations>,
) {
    // ... 复杂的阈值计算和 enabled 状态更新逻辑
    if !static_optimizations.enabled {
        return;
    }
    // ...
}

// 之后：参数为 Res<StaticTransformOptimizations>，只读。
pub fn mark_dirty_trees(
    ...,
    static_optimizations: Res<StaticTransformOptimizations>,
) {
    if !static_optimizations.is_enabled() {
        return;
    }
    // ... (移除阈值计算代码)
}

  3. 变换传播系统适配：将 static_optimizations.enabled 的检查改为 static_optimizations.is_enabled()。

    // 在 parallel 和 single_threaded 模块中，类似以下的检查都被更新：
// 之前：
if static_optimizations.enabled && !transform_tree.is_changed() { ... }

// 之后：
if static_optimizations.is_enabled() && !transform_tree.is_changed() { ... }
