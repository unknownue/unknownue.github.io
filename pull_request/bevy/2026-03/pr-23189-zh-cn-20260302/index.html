描述翻译

目标

最近添加的白炉测试没有禁用全局环境光，这导致渲染结果看起来比应有的更亮。

解决方案

在白炉测试中禁用全局环境光。

测试

  • 通过运行 cargo run --example testbed_white_furnace 进行测试。

PBR渲染器对于部分金属材质仍然会损失能量，但对于纯电介质或金属材质，在使用纯色光源时似乎工作正常。

问题与背景：被污染的“白炉”

white_furnace（白炉）测试是图形渲染，特别是PBR（Physically Based Rendering，基于物理的渲染）中一个经典的能量守恒验证场景。其核心思想很简单：将一个中性色（通常是白色）的球体放置在一个完全均匀、全方向、纯白色的环境光照下。在一个物理正确的渲染器中，球体反射出的光线总量应该等于它接收到的总量，因此渲染出的球体也应该是一个均匀的白色圆形，其亮度与环境光完全一致。如果渲染器“损失”了能量，球体就会显得比环境暗；如果“增益”了能量，球体就会更亮。这是一个非常直观的测试工具。

在Bevy中，这个测试以示例程序（example）的形式存在，允许开发者运行并目视检查PBR渲染器的正确性。然而，问题出在了测试的初始设置上。除了测试意图施加的“环境贴图光照”（environment map lighting）外，Bevy场景默认还包含一个 全局环境光（GlobalAmbientLight） 组件。这是一个简化模型，为整个场景提供一个基础的环境亮度，通常用于模拟间接光照或避免纯黑阴影。但在白炉测试的上下文中，这个额外的光源成了一个“污染源”。它使得球体接收到的总光量超过了测试预设的环境贴图所提供的光量，导致球体被渲染得比预期更亮。这掩盖了PBR渲染器可能存在的能量损失问题，使得测试失去了其诊断价值——你无法区分球体的亮度是来自正确的能量守恒反射，还是来自这个额外的全局环境光补光。

解决方案：移除干扰项

解决这个问题的思路非常直接且符合直觉：既然测试的假设是“光照应仅来自环境贴图”，那么就应该在测试场景的配置中，明确移除所有其他光源。在Bevy的ECS架构下，光照是一种资源（Resource）或组件。全局环境光通过一个名为 GlobalAmbientLight 的资源进行控制。

因此，解决方案就是在设置测试场景的系统（setup 函数）中，插入一个值为 NONE 的 GlobalAmbientLight 资源。GlobalAmbientLight::NONE 表示环境光强度为零，这有效地禁用了该光源。

实现细节：一行代码的精确修复

这个PR的改动非常简洁，只涉及一个文件的一处添加。所有逻辑都集中在场景的初始化阶段。

在 setup 函数中，在创建网格之后、开始生成一系列测试球体之前，插入了一行命令：

commands.insert_resource(GlobalAmbientLight::NONE);

这行代码利用 Commands 将 GlobalAmbientLight 资源插入到世界中，并将其值设置为 NONE。这是一个典型的Bevy ECS模式，用于在系统执行时配置世界状态。

这个修改的时机也很关键。它在球体生成之前执行，确保了之后创建的所有实体（球体）都将在“无全局环境光”的条件下被渲染。这保证了整个测试流程光照条件的一致性。

技术洞察：测试的纯粹性与渲染验证

这个PR虽然改动很小，但体现了一个重要的软件工程和测试原则：控制变量。为了有效测试某个特定模块（这里是PBR着色器的能量守恒性质），必须尽可能地隔离其他影响因素。全局环境光作为一个独立于环境贴图的光照模型，在此处是一个需要被剥离的变量。

从架构角度看，这也展示了Bevy渲染系统良好的模块化设计。不同的光照类型（环境贴图 vs 全局环境光）是独立可配置的模块，开发者可以通过添加或移除资源来灵活地组合或禁用它们，以满足像白炉测试这样的特定需求。

作者在PR描述中也提到了一个重要的观察：修复之后，测试显示PBR渲染器对于纯电介质（dielectrics，非金属）或纯金属材质，在使用纯色光源时表现正确；但对于部分金属材质，仍然存在能量损失。这实际上让测试回归了其本质功能——准确地揭示了渲染器中可能存在的问题区域，而不是用一个配置错误掩盖它们。

影响与总结

这个修复直接提升了 white_furnace 示例程序的正确性和有效性。现在，开发者运行此示例时，看到的结果更能真实反映PBR核心着色器的能量守恒性能。这有助于：

  1. 回归检测：未来对PBR着色器的任何修改，如果意外破坏了能量守恒，这个测试能更敏感地将其暴露出来。
  2. 视觉验证：为贡献者和用户提供了一个更可靠的、符合行业标准的可视化调试工具。
  3. 代码清晰：明确了该测试场景所依赖的光照假设，使代码的意图对阅读者更加清晰。

这是一个典型的“修复测试工具本身”的提交。它不直接修改渲染引擎的核心算法，而是修正了用于评估核心算法的测试夹具（fixture）的配置，从而确保了评估结果的准确性。

关键文件更改

  • examples/testbed/white_furnace.rs (+4/-0)

描述：此文件包含了白炉测试的完整场景设置逻辑。本次PR的修改是在场景初始化函数中添加了一行代码，用于将全局环境光资源设置为NONE，从而确保测试中的光照仅来源于预设的环境贴图，排除了额外环境光对测试结果的干扰。

代码片段：

// 文件：examples/testbed/white_furnace.rs
// 修改位置：在 `fn setup(...) {` 函数内

// 之前（隐含使用了默认的全局环境光）：
     let sphere_mesh = meshes.add(Sphere::new(0.45));
     // add entities to the world
     for y in -2..=2 {

// 之后（显式禁用全局环境光）：
     let sphere_mesh = meshes.add(Sphere::new(0.45));
     
     // Light should come from the environment map only
     commands.insert_resource(GlobalAmbientLight::NONE);
     
     // add entities to the world
     for y in -2..=2 {

关联性：这行代码的添加直接实现了PR的目标，是本次修复的核心所在。它通过Bevy ECS的命令系统在正确的时间点配置了渲染环境，满足了测试的预设条件。

延伸阅读

  1. 基于物理的渲染（PBR）理论：了解能量守恒、微表面模型等核心概念是理解白炉测试意义的基础。可阅读《Real-Time Rendering》第四版相关章节或LearnOpenGL - PBR Theory。
  2. 白炉测试（White Furnace Test）：这是一个在图形学文献和行业中广泛使用的测试。可以搜索“white furnace test energy conservation”来查找更多关于其原理和应用的资料。
  3. Bevy光照系统：要深入了解GlobalAmbientLight和环境贴图在Bevy中是如何工作的，可以查阅Bevy官方文档中关于bevy::pbr::AmbientLight和bevy::environment_map的模块说明。
  4. 软件测试中的夹具（Fixtures）：本次修复本质上是对测试夹具的修正。了解测试夹具的概念和最佳实践有助于编写更可靠、独立的测试。
