The Story of This Pull Request

This PR addresses a specific documentation gap in Bevy’s ECS examples. While the Bevy engine provides powerful mechanisms for registering and running systems dynamically, the exact syntax and patterns for using these features weren’t clearly demonstrated in the existing examples.

The problem was that developers needed to store and execute arbitrary logic associated with entities on-demand, rather than having systems run every frame. This pattern is particularly useful for scenarios like cutscenes, scripted events, or UI interactions that trigger specific behaviors. However, the combination of SystemId, Commands::register_system(), and Commands::run_system() can be unintuitive to discover and use correctly without concrete examples.

The solution approach was straightforward: create a clear, well-documented example that demonstrates the complete pattern. The developer recognized that showing code is more effective than describing it, so they implemented a working prototype and turned it into a proper example.

The implementation focuses on three key concepts. First, it shows how to store a reference to a system in a component using SystemId<(), ()>. This type parameterization indicates a system with no inputs and no outputs, which is typical for callback-style systems. Second, it demonstrates three different types of systems that can be registered: a simple closure with no parameters, a system with a standard query parameter, and an exclusive system that takes &mut World. This progression shows increasing complexity and capability. Third, it shows how to execute these stored systems on-demand by iterating through components and calling commands.run_system().

From a technical perspective, this example reveals several important insights about Bevy’s ECS architecture. The use of SystemId as a component allows for dynamic system composition at runtime, which is more flexible than statically defined systems. The pattern trades some performance (due to dynamic dispatch and system lookup) for flexibility, making it suitable for infrequent events rather than frame-by-frame updates. The example also hints at potential extensions, suggesting that observers could be used to detect when callbacks should run in response to entity events.

The impact of this PR is primarily educational. It provides a clear, runnable reference for developers who need to implement callback or event-driven logic in their Bevy applications. By adding this to the official examples, it reduces the learning curve for an advanced ECS pattern and ensures consistency in how developers implement these features. The example also serves as documentation for the SystemId, register_system(), and run_system() APIs, which might otherwise be overlooked.

//! Sometimes, you want an extremely flexible way to store logic associated with an entity.
//! This example demonstrates how to store arbitrary systems in components and run them on demand.
//!
//! This pattern trades some performance for flexibility and works well for things like cutscenes, scripted events,
//! or one-off UI-driven interactions that don't need to run every frame.

use bevy::{ecs::system::SystemId, prelude::*};

fn main() {
    let mut app = App::new();
    app.add_systems(Startup, setup_callbacks);
    app.add_systems(Update, run_callbacks);
    app.run();
}

#[derive(Component)]
struct Callback {
    system_id: SystemId<(), ()>,
}

fn setup_callbacks(mut commands: Commands) {
    let trivial_callback = Callback {
        system_id: commands.register_system(|| {
            println!("This is the trivial callback system");
        }),
    };

    let ordinary_system_callback = Callback {
        system_id: commands.register_system(|query: Query<&Callback>| {
            let n_callbacks = query.iter().len();
            println!("This is the ordinary callback system. There are currently {n_callbacks} callbacks in the world.");
        }),
    };

    let exclusive_callback = Callback {
        system_id: commands.register_system(|world: &mut World| {
            let n_entities = world.entities().len();
            println!("This is the exclusive callback system. There are currently {n_entities} entities in the world.");
        }),
    };

    commands.spawn(trivial_callback);
    commands.spawn(ordinary_system_callback);
    commands.spawn(exclusive_callback);
}

// In many cases, you might want to use an observer to detect when a callback should run,
// triggering the callback in response to some entity-event!
fn run_callbacks(mut commands: Commands, query: Query<&Callback>) {
    for callback in query.iter() {
        commands.run_system(callback.system_id);
    }
}
