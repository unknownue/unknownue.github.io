分析报告：修复法线接近-Z时sample_visible_ggx的不稳定性问题

基本信息

  • 标题: Fix instability in sample_visible_ggx for normals near -Z
  • PR链接: https://github.com/bevyengine/bevy/pull/23194
  • 作者: dylansechet
  • 状态: 已合并

描述翻译

目标

sample_visible_ggx函数中执行了normal.z + 1的除法操作，当输入法线接近(0, 0, -1)时会产生无效值。这导致GeneratedEnvironmentMapLight中出现伪影，在白炉测试中可见。

解决方案

采用https://bruop.github.io/ibl中描述的方法，当法线接近±Z时使用带有特殊情况的TBN矩阵，而不是当前实现的反射方法。

测试

  • 在main分支和我的分支上运行cargo run --example reflection_probes，确保没有明显差异
  • 运行cargo run --example testbed_white_furnace，检查伪影是否消失

本次PR的技术分析

问题背景与根本原因

在物理渲染中，sample_visible_ggx函数用于生成环境光照采样，是图像光照(Image-based Lighting, IBL)系统的关键组件。原实现存在一个数值稳定性问题：当表面法线接近负Z轴方向(0, 0, -1)时，代码中的除法操作wr.z可能产生极大的值或NaN。

问题出现在以下代码段：

let wr = n + up;  // up = vec3f(0.0, 0.0, 1.0)
let c = dot(wr, c_std) * wr / wr.z - c_std;

当n接近(0, 0, -1)时，wr.z趋近于0，导致除法结果趋于无穷大，进而产生渲染伪影。这种边缘情况在球形表面渲染时尤其明显，特别是环境贴图采样中。

技术解决方案

作者采用了来自bruop.github.io/ibl的技术方案，将原来的反射方法替换为基于TBN矩阵的正交化方法。

原方法试图通过反射操作将采样向量对齐到+Z轴，但存在数值不稳定问题。新方法使用orthonormalize函数构建一个切线-副切线-法线(TBN)矩阵，然后通过矩阵乘法旋转采样向量。

实现细节与代码分析

核心改动在crates/bevy_pbr/src/render/pbr_lighting.wgsl文件中：

  1. 导入语句更新：新增对orthonormalize函数的导入
// 之前：
#import bevy_render::maths::PI

// 之后：
#import bevy_render::maths::{PI, orthonormalize}
  1. 采样算法重构：完全替换了向量对齐逻辑
// 之前的不稳定实现：
// Reflect the sample so that the normal aligns with +Z
let up = vec3f(0. 0, 0.0, 1.0);
let wr = n + up;
let c = dot(wr, c_std) * wr / wr.z - c_std;

// 之后的稳定实现：
// Rotate the sample so that the normal aligns with +Z
let TBN = orthonormalize(n);
let c = TBN * c_std;

新实现的关键优势：

  • 数值稳定性：避免除以接近零的值
  • 数学正确性：使用正交化矩阵确保旋转操作保持向量长度不变
  • 代码简洁性：逻辑更清晰，减少了计算步骤

正交化方法的技术说明

orthonormalize(n)函数构造一个3×3正交矩阵，其中第三列（在WGSL中可能是第三行，取决于存储布局）是归一化的输入法线n。这个矩阵将一个以法线为Z轴的正交坐标系映射到标准坐标系(0,0,1)。

这种方法的优势在于：

  1. 避免了原反射方法中分母接近零的问题
  2. 提供了一种更通用的坐标变换框架
  3. 在数学上更稳定，特别是在法线接近极点的边缘情况下

测试验证与质量保证

作者执行了系统性测试来验证修复效果：

  1. 功能回归测试：通过reflection_probes示例确保核心功能不受影响
  2. 问题专项测试：使用testbed_white_furnace示例专门验证伪影是否消除

白炉测试对比图清楚展示了修复效果：修复前的图像在法线接近-Z区域出现明显的亮斑和噪声，而修复后的图像表现出均匀的辐射分布，符合物理正确性。

工程意义与最佳实践

这个修复展示了几个重要的工程实践：

  1. 边缘情况处理：渲染算法必须考虑所有可能的输入范围，特别是数值不稳定的边界条件
  2. 数值稳定性优先：在计算机图形学中，数值稳定性通常比理论简洁性更重要
  3. 引用外部权威资源：作者引用了行业认可的IBL实现方案，增加了解决方案的可信度
  4. 最小化修改：修复仅涉及必要的代码变更，保持了系统的简洁性

潜在影响与后续考虑

虽然这个修复解决了当前问题，但在更广泛的上下文中需要考虑：

  1. 性能影响：orthonormalize函数可能比原来的反射操作更复杂，需要评估性能变化
  2. GPU兼容性：正交化算法在不同GPU架构上的表现一致性
  3. 扩展性：新的TBN方法为后续的渲染优化提供了更灵活的框架
