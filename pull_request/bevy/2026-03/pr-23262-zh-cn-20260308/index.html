本次 Pull Request 的故事

这个 PR 源于一个非常具体且常见的需求：让 EasingCurve 类型支持相等性比较。在使用 Bevy 进行动画或缓动效果开发时，开发者可能需要检查两个缓动曲线是否相同，例如在单元测试中验证行为，或在运行时根据缓动曲线类型做出不同决策。

问题 #23246 直接指出了 EasingCurve 结构体缺少 PartialEq trait 实现。在 Rust 中，PartialEq trait 是启用 == 和 != 操作符的基础。如果一个类型没有实现这个 trait，就无法直接进行相等性比较，这会给开发者带来不便。

解决这个问题的方法非常直接。EasingCurve 本身是一个泛型结构体，其内部字段理论上都可以进行比较（只要其泛型参数 T 也实现了 PartialEq）。因此，最合理的方案是修改 EasingCurve 的定义，在 derive 属性中添加 PartialEq。这个改动很小，但非常有效。

具体的实现就是修改一行代码。在 crates/bevy_math/src/curve/easing.rs 文件中，EasingCurve 结构体的定义从：

#[derive(Clone, Debug)]

改为：

#[derive(Clone, Debug, PartialEq)]

这个改动遵循了 Rust 的一个常见模式：当为一个泛型结构体派生 PartialEq 时，编译器会自动要求该结构体的所有字段（包括泛型参数 T 所代表的类型）都实现 PartialEq。如果 T 没有实现 PartialEq，那么 EasingCurve<T> 也不会实现 PartialEq。这是一种类型安全的保证。

从技术角度看，这个改动是向后兼容的。它只是为 EasingCurve 增加了一个 trait 实现，没有改变任何现有行为。只要代码原本能编译，现在依然能编译。同时，它也没有引入性能开销，因为 PartialEq 的派生实现是编译时生成的，运行时只是简单的字段比较。

这个 PR 展示了开源协作中一个典型的“修复小问题”的场景。问题明确，解决方案直接，影响范围小但实用。它不需要复杂的测试，因为 Rust 的 trait 系统保证了派生实现的正确性。合并这样的 PR 有助于提高代码库的完整性和开发者的使用体验。

关键文件变更

  • crates/bevy_math/src/curve/easing.rs (+1/-1)

这个文件包含了 EasingCurve 结构体的定义。改动只有一行：在 derive 属性列表中添加了 PartialEq。

变更详情：

// 文件: crates/bevy_math/src/curve/easing.rs
// 变更前:
#[derive(Clone, Debug)]
#[cfg_attr(feature = "serialize", derive(serde::Serialize, serde::Deserialize))]
#[cfg_attr(feature = "bevy_reflect", derive(bevy_reflect::Reflect))]
pub struct EasingCurve<T> {
    // ... 字段定义
}

// 变更后:
#[derive(Clone, Debug, PartialEq)] // 添加了 PartialEq
#[cfg_attr(feature = "serialize", derive(serde::Serialize, serde::Deserialize))]
#[cfg_attr(feature = "bevy_reflect", derive(bevy_reflect::Reflect))]
pub struct EasingCurve<T> {
    // ... 字段定义保持不变
}

这个改动直接解决了 issue 中描述的问题，使 EasingCurve 实例现在可以使用 == 和 != 操作符进行比较。
