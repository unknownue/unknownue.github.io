如何准备一次发布

[!注意]
仅对具有写入权限的维护者相关。

  1. 从 main 分支切换到一个新分支。
  2. 运行 npm test 以确保所有测试通过。
  3. 更新 https://github.com/actions/cache/blob/main/package.json 中的版本号。
  4. 运行 npm run build 以更新编译后的文件。
  5. 使用新版本和变更内容更新此 https://github.com/actions/cache/blob/main/RELEASES.md 文件中的 ## 更新日志 部分。
  6. 运行 licensed cache 以更新许可证报告。
  7. 运行 licensed status 并通过更新 https://github.com/actions/cache/blob/main/.licensed.yml 文件来解析所有警告。
  8. 提交您的更改并将您的分支推送到上游。
  9. 针对 main 分支开启一个 Pull Request 并等待审核和合并。
  10. 在 https://github.com/actions/cache/releases 起草一个新版本，使用与 package.json 相同的版本号
    1. 创建一个带有该版本号的新标签。
    2. 自动生成版本说明，并根据您在 RELEASES.md 中所做的更改进行更新。
    3. 切换设置为最新版本的选项。
    4. 发布该版本。
  11. 导航至 https://github.com/actions/cache/actions/workflows/release-new-action-version.yml
    1. 应该有一个相同版本号的待执行工作流。
    2. 批准此次运行以发布新版本并更新此 Action 的主要标签。

本次 PR 的故事

这是一个由 Dependabot 自动创建的常规依赖更新 PR，目的是将 Bevy 项目中的 GitHub Actions actions/cache 从版本 5.0.1 更新到 5.0.3。对于像 Bevy 这样的大型 Rust 游戏引擎项目，保持 CI/CD 流水线中依赖项的最新状态是一项重要的维护工作。

背景与问题 Bevy 的持续集成（CI）流水线广泛使用 actions/cache 来缓存 Cargo（Rust 的包管理器）的构建产物。这能显著减少构建时间，因为后续的流水线运行可以直接复用之前已编译的依赖项缓存，而无需每次都从零开始编译所有内容。然而，依赖项会不断更新，新版本可能包含重要的安全补丁、错误修复或性能改进。使用过时版本可能导致已知的安全漏洞未被修复，或者错过对 CI 稳定性和效率的改进。

解决方案与实施 此次更新的解决方案非常简单直接：将项目各个工作流（workflow）文件中所有对 actions/cache/restore 和 actions/cache/save 的引用，从特定的旧提交哈希 9255dc7a253b0ccc959486e2bca901246202afeb （对应 v5.0.1）更新到新提交哈希 cdf6c1fa76f9f475f3d7449005a359c84ca0f306 （对应 v5.0.3）。通过查看更新日志，我们可以看到这次升级的主要目的：

  1. 在 v5.0.3 中，升级了 @actions/cache 到 v5.0.5，并解决了编号为 #33 的安全依赖问题。
  2. 在 v5.0.2 中，修复了当创建缓存条目时，对来自缓存服务的 429（请求过多）状态码不再进行重试的行为。这有助于避免在服务受限时产生不必要的负载或延迟。

Dependabot 自动完成了这一查找和替换工作。维护者只需要审查变更内容，确认新版本没有引入破坏性变更，并通过 CI 测试即可。

技术洞察 虽然这个 PR 看起来只是改了几个版本号，但它背后体现了现代软件开发中依赖管理的最佳实践。使用固定提交哈希（如 @cdf6c1fa...），而不是标签（如 @v5），能确保流水线的完全可复现性，避免因标签被意外移动而导致构建行为变化。Dependabot 这样的自动化工具对于维护大型项目的众多依赖项至关重要，它能及时提醒并应用安全更新，降低维护负担和潜在风险。

影响 此次更新对 Bevy 项目的主要影响是：

  • 安全性提升：通过升级子依赖 @actions/cache 来修复已知安全漏洞（Dependabot issue #33），增强了 CI 流水线基础设施的安全性。
  • 行为微调：对 429 错误的重试行为调整，理论上可以使 CI 在缓存服务繁忙时表现得更合理。
  • 维持更新：保持项目与上游依赖的同步，以便未来能顺利接收更多更新和修复。

这是一个低风险、高收益的维护性更新。变更范围仅限于 .github/workflows/ 目录下的 CI 配置文件，不涉及任何产品代码，因此在 CI 测试通过后可以安全合并。
