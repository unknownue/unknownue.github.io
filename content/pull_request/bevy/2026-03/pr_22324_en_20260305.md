## Description Translation
# Objective

Fix `non_camel_case_types` warning emitted by generic type parameters generated by `#[system_param(builder)]`.

```rust
#[derive(SystemParam)]
#[system_param(builder)]
struct MySystemParam<'w, 's> {
    query_field: Query<'w, 's ,Entity>,
}
```

```
> cargo check
...
warning: type parameter `Bquery_field` should have an upper camel case name
   --> src\lib.rs:171:9
    |
171 |         query_field: Query<'w, 's ,Entity>,
    |         ^^^^^^^^^^^ help: convert the identifier to upper camel case: `BqueryField`
...
```

## Solution

Add `#[allow(non_camel_case_types)]` to each generated type parameter when declaring, in the `SystemParam` derive macro.

## Testing

By referencing this branch in Cargo.toml, the warning no longer occurs for the usage above when running `cargo check`.

## The Story of This Pull Request

The issue started when a developer used Bevy's `#[derive(SystemParam)]` macro with the `#[system_param(builder)]` attribute. When they defined a system parameter struct with snake_case field names, the macro generated type parameters for the builder that matched these field names, but with a "B" prefix. This naming convention triggered Rust's `non_camel_case_types` lint, which expects type parameters to use UpperCamelCase.

The problem occurred because the macro generated type parameters like `Bquery_field` for a field named `query_field`. The lint warning was technically correct - these type parameters should be `BqueryField` in UpperCamelCase. However, changing the naming convention would break the macro's internal logic and potentially affect user code that depends on these generated names.

The developer considered two main approaches to solve this:
1. Change the macro to generate UpperCamelCase type parameters
2. Add `#[allow(non_camel_case_types)]` attributes to suppress the warnings

The first approach would be more correct from a Rust naming conventions perspective, but it would require more extensive changes. The generated type parameters are used internally by the macro system, and changing their naming convention could potentially break downstream code or require updates to the macro's implementation logic. The second approach, while technically allowing non-conventional names, is simpler and safer since it only affects lint warnings without changing any actual functionality.

The chosen solution adds `#[allow(non_camel_case_types)]` attributes to the generated type parameters in two places within the `SystemParam` derive macro implementation. This approach acknowledges that the generated names come from user-defined field names (which are typically in snake_case by Rust convention), and that automatically converting them to UpperCamelCase could introduce complexity or unexpected behavior.

The implementation modifies the macro's code generation in `crates/bevy_ecs/macros/src/lib.rs`. The changes are minimal and focused - only two lines are modified, each adding the same attribute annotation. This maintains the existing naming scheme while preventing the lint warning from appearing in user code. The `reason` parameter in the attribute provides documentation explaining why the non-standard naming is acceptable in this context.

The technical insight here is that procedural macros in Rust often generate identifiers based on user input, and sometimes these generated identifiers don't follow Rust's standard naming conventions. In such cases, it's acceptable to suppress the lint warnings rather than trying to enforce conventions on generated code. This is particularly true when the generated identifiers are internal to the macro system and not meant to be used directly by developers.

The impact of this change is straightforward: developers using the `#[system_param(builder)]` attribute no longer see false positive lint warnings about type parameter naming. This improves the developer experience by reducing noise during compilation without affecting the functionality or performance of the generated code.

From an engineering perspective, this fix demonstrates good practice in macro design - when generating code that might trigger lints, it's considerate to add appropriate allow attributes. This keeps the focus on actual issues in user code rather than artifacts of code generation.

## Key Files Changed

**File: `crates/bevy_ecs/macros/src/lib.rs`**

This file contains the procedural macro implementation for deriving `SystemParam` traits. The changes specifically affect the code generation for the builder pattern when the `#[system_param(builder)]` attribute is used.

The key changes are in the `derive_system_param_impl` function where the builder struct and its implementation are generated:

```rust
// Before (line 384-387):
let builder_struct = quote! {
    #[doc = #builder_doc_comment]
    struct #builder_name<#(#builder_type_parameters,)*> {
        #(#field_members: #builder_type_parameters,)*
    }
};

// After:
let builder_struct = quote! {
    #[doc = #builder_doc_comment]
    struct #builder_name<#(#[allow(non_camel_case_types, reason = "generated from snake-case field name")] #builder_type_parameters,)*> {
        #(#field_members: #builder_type_parameters,)*
    }
};
```

```rust
// Before (line 394-400):
// SAFETY: This delegates to the `SystemParamBuilder` for tuples.
unsafe impl<
    #(#lifetimes,)*
    #(#builder_type_parameters: #path::system::SystemParamBuilder<#field_types>,)*
    #punctuated_generics
> #path::system::SystemParamBuilder<#generic_struct> for #builder_name<#(#builder_type_parameters,)*>
    #where_clause

// After:
// SAFETY: This delegates to the `SystemParamBuilder` for tuples.
unsafe impl<
    #(#lifetimes,)*
    #(#[allow(non_camel_case_types, reason = "generated from snake-case field name")] #builder_type_parameters: #path::system::SystemParamBuilder<#field_types>,)*
    #punctuated_generics
> #path::system::SystemParamBuilder<#generic_struct> for #builder_name<#(#builder_type_parameters,)*>
    #where_clause
```

Both changes add the same attribute `#[allow(non_camel_case_types, reason = "generated from snake-case field name")]` to each generated type parameter. This tells the Rust compiler to suppress the `non_camel_case_types` lint for these specific type parameters, with a clear explanation of why the exception is needed.
