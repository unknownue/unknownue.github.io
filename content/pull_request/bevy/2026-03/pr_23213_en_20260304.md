## The Story of This Pull Request

This PR addresses a straightforward code quality improvement in the Bevy game engine's PBR (Physically Based Rendering) clustering system. The issue was a custom rounding function that duplicated functionality already available in the Rust standard library.

In the `crates/bevy_pbr/src/cluster/gpu.rs` file, there was a helper function called `round_up` that rounded a given value up to the nearest multiple of 32. This function used bitwise operations: `(length + 31) & !31`. While this implementation is correct and efficient, it represents a case of "Not Invented Here" (NIH) syndrome - reinventing functionality that already exists in the standard library.

The developer recognized that Rust's integer types have a built-in method `next_multiple_of()` that provides exactly this functionality. This method was stabilized in Rust 1.73.0 and offers a more readable, maintainable approach to the same operation.

The implementation change is minimal but meaningful. In the `prepare_cluster_dummy_textures` function, two calls to `round_up()` were replaced with direct calls to `.next_multiple_of(32)`. The custom helper function was then removed entirely since it was no longer needed.

This change improves code maintainability in several ways:
1. **Standard library usage**: Using standard library functions reduces cognitive load for other developers who are familiar with Rust's API
2. **Readability**: `next_multiple_of()` clearly expresses the intent, whereas the bitwise operations require mental parsing
3. **Future-proofing**: If the standard library implementation changes or improves, the code automatically benefits
4. **Reduced code surface**: Removing a helper function reduces the amount of code to maintain and test

The PR is labeled as "D-Trivial" which is appropriate given the small scope of the change. However, even trivial changes like this contribute to overall codebase health by eliminating redundant implementations and promoting consistent use of standard library features.

From an architectural perspective, this change doesn't affect the rendering logic or performance - it's purely a code quality improvement. The texture dimensions are still rounded up to the nearest multiple of 32 to meet alignment requirements for GPU operations, which is important for performance and compatibility. The standard library's `next_multiple_of()` method likely uses similar bitwise operations internally, so there should be no performance regression.

This type of refactoring is valuable in a large codebase like Bevy because it makes the code more approachable for new contributors and easier to maintain for the core team. It's a good example of how small, focused improvements can collectively improve a project's overall quality.

## Key Files Changed

**File:** `crates/bevy_pbr/src/cluster/gpu.rs` (+2/-7 lines)

**What changed:** Replaced a custom rounding function with the standard library's `next_multiple_of()` method.

**Code snippets:**

```rust
// Before:
size: Extent3d {
    width: round_up(view_cluster_config.dimensions.x),
    height: round_up(view_cluster_config.dimensions.y),
    depth_or_array_layers: 1,
},

// Helper function at the end of prepare_cluster_dummy_textures:
fn round_up(length: u32) -> u32 {
    (length + 31) & !31
}
```

```rust
// After:
size: Extent3d {
    width: view_cluster_config.dimensions.x.next_multiple_of(32),
    height: view_cluster_config.dimensions.y.next_multiple_of(32),
    depth_or_array_layers: 1,
},

// Helper function removed entirely
```

**Relationship to PR purpose:** This change directly implements the PR's objective of avoiding NIH (Not Invented Here) by replacing custom rounding logic with a standard library equivalent. The code is functionally identical but more maintainable and idiomatic.
