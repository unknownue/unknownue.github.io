## The Story of This Pull Request

这个PR的故事始于一个例行维护任务：Dependabot 检测到项目依赖的 GitHub Action `actions/upload-artifact` 有新的主版本（major version）v7.0.0 可用，并自动创建了升级PR。对于像 Bevy 这样活跃的项目，保持 CI/CD 流水线中工具的更新至关重要，这不仅能获得新功能，还能确保安全性和兼容性。

`actions/upload-artifact` 在 Bevy 的 GitHub Actions 工作流中扮演着关键角色。它的核心功能是在 CI 运行结束后，将生成的文件（如构建日志、测试截图、性能追踪文件等）打包并上传到 GitHub，供开发者下载和分析。这在调试 CI 失败或审查示例的视觉输出时非常有用。在本次升级前，Bevy 使用的是 v6.0.0 版本。

查看 v7.0.0 的发布说明，这次升级带来了两个主要变化。首先是 **“直接上传”（Direct Uploads）** 功能。在 v6 及更早的版本中，`upload-artifact` 总是会将匹配的文件或目录压缩成一个 ZIP 包，然后上传。v7 引入了 `archive: false` 参数，允许跳过压缩步骤，直接将单个文件上传。这带来的好处是，当上传产物只是一个文件（比如一张截图或一个 JSON 追踪文件）时，可以减少一次压缩操作，加快上传速度，并且下游使用者下载后也无需解压，直接可用。对于 Bevy 工作流中上传的 `trace.json` 和 `screenshot-*.png` 文件，这个功能具有潜在价值。

第二个主要变更是 **向 ESM 模块的迁移**。`@actions` 工具包的其他依赖正在转向 ESM（ECMAScript Modules），这要求 `upload-artifact` 也必须升级以适应其依赖生态。这是一个底层的技术栈升级，对使用者来说通常是透明的，但它是维护项目长期健康、避免因依赖过时而导致安全或功能问题的必要步骤。

那么，升级一个主要版本，尤其是像这样一个被广泛使用的核心 Action，风险有多大？这通常是维护者最关心的问题。从 Dependabot 提供的兼容性分数（Compatibility Score）来看，这次升级是兼容的。更重要的是，我们需要审视 Bevy 是如何使用这个 Action 的。浏览变更的代码（diff）可以发现，所有对 `upload-artifact` 的调用都遵循相同的简单模式：指定一个 SHA 版本和上传的路径。Bevy 没有使用任何在 v7 中可能被移除或行为发生改变的复杂参数或高级功能。它的使用场景非常直接：`if` 条件触发后，将某个目录下的文件上传并命名为一个产物。因此，此次升级的风险被评估为很低。

基于上述分析，维护者 `mockersf` 合并了这个 PR。整个升级过程完全是机械式的：在四个不同的工作流 YAML 文件（`.github/workflows/ci.yml`, `.github/workflows/example-run.yml`, `.github/workflows/example-run-report.yml`, `.github/workflows/validation-jobs.yml`）中，共 9 处使用 `actions/upload-artifact` 的地方，将其引用从 v6.0.0 对应的旧提交 SHA `b7c566a772e6b6bfb58ed0dc250532a479d7789f` 更新为 v7.0.0 对应的新提交 SHA `bbbca2ddaa5d8feaa63e36b76fdaad77386f024f`。通过锁定具体的提交 SHA 而非标签，确保了工作流行为的确定性和可复现性。

从技术角度看，这次升级本身没有改变 Bevy 的任何逻辑代码，但它是软件工程中依赖管理（dependency management）的一个标准实践案例。它展示了如何通过自动化工具（Dependabot）和安全评估（兼容性分数、代码审查）来平稳地推进基础设施的迭代。升级后，Bevy 的 CI 流水线立即获得了对新功能的支持（尽管当前配置尚未启用直接上传），并使其底层工具链与生态系统保持同步，为未来的维护扫清了障碍。

## Key Files Changed

所有更改的文件都在 `.github/workflows/` 目录下，它们定义了 Bevy 项目的持续集成和部署流程。

1.  **`.github/workflows/ci.yml`**
    *   **作用**：定义了主要的 CI 流程，包括检查、测试、构建等任务。修改点出现在上传“missing-examples”、“missing-features”和“msrv”产物的步骤中，这些步骤仅在特定任务失败时触发，用于收集调试信息。
    *   **关键修改**：将 `upload-artifact` 的版本引用 SHA 更新至 v7.0.0。
    ```yaml

    ```

2.  **`.github/workflows/example-run.yml`**
    *   **作用**：在不同的操作系统（macOS, Linux, Windows）上运行 Bevy 示例，并上传生成的性能追踪文件（traces）和截图（screenshots）以供检查。此外，如果整个示例运行任务失败，会上传整个工作区（workspace）的产物用于调试。
    *   **关键修改**：文件中多处用于上传 traces、screenshots 和完整工作区的步骤都更新了 `upload-artifact` 的版本。这是修改行数最多的文件。
