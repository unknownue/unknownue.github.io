## Description Translation

# 目标

我注意到更改 `AlphaMode::Mask` 的阈值对我的材质没有效果。查看PBR着色器后，我发现它只在读取光照相关属性时读取 `alpha_cutoff`，并且只针对受光照（lit）的材质。

对于不受光照（unlit）的材质，它回退到了 [0.5 的默认值](https://github.com/bevyengine/bevy/blob/main/crates/bevy_pbr/src/render/pbr_types.wgsl#L86)。

## 解决方案

将 `alpha_cutoff` 的读取移出仅针对受光照材质的代码块。

## 测试

transparency_3d 示例中有一个使用 `AlphaMode::Mask` 的 unlit 球体。我将它的阈值从 0.5 改为 0.1，以使差异可见（见下文）。

_我不确定是否应该包含示例的更改，但我觉得展示 Mask 模式可能是有益的。_

## The Story of This Pull Request

这个 PR 始于开发者 `doceazedo` 在项目中遇到的一个直观问题：他为一个 unlit 材质（不受光照影响的材质）设置了 `AlphaMode::Mask` 并尝试调整其 `threshold`（阈值），却发现无论怎样调整，渲染结果都没有任何变化。这是一个典型的 API 行为与预期不符的 bug，对于依赖精确透明度控制（例如制作 UI 遮罩、镂空效果等）的项目来说，会直接导致功能失效。

为了定位问题，开发者深入查看了 Bevy 的物理渲染（PBR，Physically Based Rendering）着色器代码。问题核心在 `pbr_fragment.wgsl` 文件中的 `pbr_input_from_standard_material` 函数里。这个函数负责从材质数据中提取所有必要信息，并组装成供后续光照计算使用的 `pbr_input` 结构体。在原始的代码逻辑中，多个材质属性的读取被组织在一个条件判断块内：

```wgsl
// 原始逻辑（简化示意）：
if (material_is_LIT) {
    // 读取光照相关属性，例如 ior, attenuation_color, attenuation_distance
    // 同时也读取了 alpha_cutoff
    pbr_input.material.alpha_cutoff = /* ... */;
}
```

这段代码的逻辑暗示了一个设计假设：`alpha_cutoff`（透明度裁切阈值）被认为是与光照计算紧密相关的属性。然而，`AlphaMode::Mask` 的工作机制实际上独立于光照模型。它根据材质的基础颜色（base_color）的 alpha 通道值与 `alpha_cutoff` 阈值进行比较，决定当前像素是完全透明（丢弃）还是完全不透明。无论材质是否受光照（lit 或 unlit），这个裁切操作都应该发生。

因此，将 `alpha_cutoff` 的读取和赋值放在 `if (material_is_LIT)` 块内，就导致了一个 bug：当材质标记为 `unlit`（即 `STANDARD_MATERIAL_FLAGS_UNLIT_BIT` 被设置）时，这个条件块被跳过，`pbr_input.material.alpha_cutoff` 根本没有被赋值。根据 WGSL 的规则，未显式初始化的结构体成员会保持其默认值。在 `pbr_types.wgsl` 中，`alpha_cutoff` 的默认值恰好是 `0.5`。这就是为什么开发者观察到阈值调整无效，材质始终以 0.5 的固定阈值进行裁切。

解决方案非常直接，但需要精确理解数据流。开发者并没有改变 `alpha_cutoff` 的存储位置或类型，而是调整了它在数据提取流水线中的顺序。具体操作分为两步：
1. **提前读取**：将 `alpha_cutoff` 的读取操作从条件块内部移到函数开头、条件判断之前的位置。这样，无论后续判断结果如何，`alpha_cutoff` 变量都已经从材质绑定数据中获得了正确的值。
2. **无条件赋值**：将 `pbr_input.material.alpha_cutoff = alpha_cutoff;` 这条赋值语句也移到条件块外部，确保 `pbr_input` 结构体中的这个字段在任何情况下都会被正确设置。

这个修复虽然代码改动量很小（仅移动了 6 行代码），但纠正了一个重要的逻辑错误，确保了 `AlphaMode::Mask` 在所有材质类型上行为的一致性。它凸显了在渲染管线中，数据准备阶段（如 `pbr_input_from_standard_material`）必须严谨，确保下游所有阶段（如执行 `AlphaMode::Mask` 裁切的片段着色器代码）所需的数据完整且正确，而不管某些高级特性（如光照）是否启用。

为了验证修复效果，开发者更新了 `examples/3d/transparency_3d.rs` 示例。他将示例中 unlit 球体的遮罩阈值从默认的 `0.5` 改为 `0.1`。在修复前，由于阈值的改变被忽略，unlit 球体与 lit 球体（阈值也为 0.5）会同时出现在视野中。修复后，unlit 球体的阈值 `0.1` 生效，意味着 alpha 值低于 0.1 的像素被丢弃，球体在更早的阶段（即旋转到更“薄”的角度时）就变得不可见，这与提供的对比视频完全吻合。

这个 PR 展示了一个典型的渲染 bug 排查和修复过程：从观察到的错误现象出发，通过阅读底层着色器代码理解数据流和逻辑，定位到不正确的条件依赖，最后通过调整代码结构来确保数据的完整性。它提醒我们，在实现基于标志位（flag）的条件逻辑时，需要仔细审查每个操作是否真的应该受到该标志位的约束。

## Key Files Changed

### 1. `crates/bevy_pbr/src/render/pbr_fragment.wgsl`
**修改原因**：修复 `alpha_cutoff` 属性仅对 lit 材质生效，而对 unlit 材质被忽略的 bug。核心改动是将 `alpha_cutoff` 的读取和赋值逻辑移出仅针对 lit 材质的条件判断块。

**关键代码变更**：
```wgsl
// 修复前（相关代码片段）:
#ifdef BINDLESS
    // ... 其他属性 ...
    // alpha_cutoff 在后续的 lit-only 块内读取
#else
    // ... 其他属性 ...
    // alpha_cutoff 在后续的 lit-only 块内读取
#endif

// ... 其他赋值 ...

if ((flags & pbr_types::STANDARD_MATERIAL_FLAGS_UNLIT_BIT) == 0u) {
    // 只有 lit 材质进入此块
#ifdef BINDLESS
        pbr_input.material.alpha_cutoff = pbr_bindings::material_array[...].alpha_cutoff;
#else
        pbr_input.material.alpha_cutoff = pbr_bindings::material.alpha_cutoff;
#endif
}
```

```wgsl
// 修复后:
#ifdef BINDLESS
    let base_color = pbr_bindings::material_array[material_indices[slot].material].base_color;
    let deferred_lighting_pass_id = pbr_bindings::material_array[material_indices[slot].material].deferred_lighting_pass_id;
    let alpha_cutoff = pbr_bindings::material_array[material_indices[slot].material].alpha_cutoff; // 移动到此：提前读取
#else
    let base_color = pbr_bindings::material.base_color;
    let deferred_lighting_pass_id = pbr_bindings::material.deferred_lighting_pass_id;
    let alpha_cutoff = pbr_bindings::material.alpha_cutoff; // 移动到此：提前读取
#endif

// ... 其他赋值 ...

pbr_input.material.flags = flags;
pbr_input.material.alpha_cutoff = alpha_cutoff; // 移动到此：无条件赋值

if ((flags & pbr_types::STANDARD_MATERIAL_FLAGS_UNLIT_BIT) == 0u) {
    // 此块内现在只包含真正依赖光照的属性 (ior, attenuation_color等)
    // alpha_cutoff 相关的行已被移除
}
```

**与PR目的的关系**：这是实现修复的核心代码变更。通过将 `alpha_cutoff` 的处理与光照判断解耦，确保了 unlit 材质也能使用自定义的遮罩阈值。

### 2. `examples/3d/transparency_3d.rs`
**修改原因**：更新示例以更清晰地展示修复效果。将 unlit 球体的遮罩阈值从 0.5 改为 0.1，使得修复前后（unlit球体出现时机）的差异更加明显，从而验证了bug已被修复。

**关键代码变更**：
```rust
// 修复前:
alpha_mode: AlphaMode::Mask(0.5),

// 修复后:
alpha_mode: AlphaMode::Mask(0.1),
```

**与PR目的的关系**：这是一个辅助性的、用于演示和验证的更改。它本身不是 bug 修复的一部分，但通过提供一个视觉上可验证的测试用例，增强了 PR 的说服力，并帮助审查者理解修复的实际影响。
