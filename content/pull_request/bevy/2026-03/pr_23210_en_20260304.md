## Description Translation

# Objective

`bevy_platform` optionally uses `wasm-bindgen-futures/std`, but that feature did not exist before version 0.4.46. This can result in a build failure when adding a Bevy dependency to a project with a long-standing lockfile or intentionally minimal versions.

## Solution

Increase the minimum version requirement for `bevy_platform` → `wasm-bindgen-futures` to be 0.4.46, not 0.4.

## Testing

Not tested beyond CI, but it would be difficult for this change to break anything.

(Testing for bugs of the type this PR fixes is #9593.)

## The Story of This Pull Request

This pull request addresses a dependency version mismatch issue in Bevy's WebAssembly support. The problem occurs in `bevy_platform`, which is Bevy's platform abstraction layer that handles different platform implementations, including WebAssembly targets.

When Bevy targets WebAssembly, it uses the `wasm-bindgen-futures` crate to bridge JavaScript's Promise API with Rust's async/await system. The `bevy_platform` crate conditionally enables the `std` feature of `wasm-bindgen-futures` for WebAssembly targets, but this feature was only introduced in version 0.4.46 of `wasm-bindgen-futures`.

The issue manifests when developers use Bevy in projects with older dependency lockfiles or when using minimal version selection (the `-Z minimal-versions` Cargo flag). In these scenarios, Cargo might resolve `wasm-bindgen-futures` to a version earlier than 0.4.46, which doesn't have the `std` feature. This causes a build failure because Bevy tries to enable a feature that doesn't exist in the resolved version.

The fix is straightforward: update the version constraint in `Cargo.toml` from `"0.4"` to `"0.4.46"`. This ensures that any version of `wasm-bindgen-futures` resolved will be at least 0.4.46, which includes the `std` feature that Bevy requires.

This type of issue is common in Rust projects that use Cargo features. When a crate adds a new feature in a patch version, downstream crates that depend on that feature need to update their minimum version constraints accordingly. The challenge is that Cargo's default version resolution (caret requirements like `^0.4`) allows any 0.4.x version, but if the feature was added in 0.4.46, versions 0.4.0 through 0.4.45 won't have it.

The PR author notes that testing for this specific type of bug is tracked in issue #9593, which suggests there's ongoing work to improve testing for dependency version compatibility issues in Bevy's CI pipeline. This is important because while the change is simple, catching these issues automatically can prevent build failures for users.

## Key Files Changed

### `crates/bevy_platform/Cargo.toml`

This file contains the dependency specifications for the `bevy_platform` crate. The change updates the version constraint for the `wasm-bindgen-futures` dependency to ensure it includes the `std` feature that Bevy requires.

**Before:**
```toml
wasm-bindgen-futures = { version = "0.4", default-features = false, optional = true }
```

**After:**
```toml
wasm-bindgen-futures = { version = "0.4.46", default-features = false, optional = true }
```

The change modifies only the version string from `"0.4"` to `"0.4.46"`. This is a targeted fix that ensures backward compatibility while preventing build failures with older versions of the dependency.
