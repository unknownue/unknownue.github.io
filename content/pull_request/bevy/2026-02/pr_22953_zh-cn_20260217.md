## Description Translation
# 目标 (Objective)

- 目前，`FreeCamera` 使用 `MouseButton::Left` 来捕获鼠标光标，这存在输入冲突的风险。例如，如果未来引入了场景选择工具，两个操作将映射到同一个触发器上。
- 将其重新映射到 `MouseButton::Right` 符合行业标准的交互范式（例如 Unity, Unreal, Blender），在这些范式中，右键专门用于视图操作，左键则专门用于对象选择和 UI 交互。

## 解决方案 (Solution)

- 将 `FreeCamera` 的鼠标捕获按钮从左键改为右键。

## 测试 (Testing)

- `free_camera_controller` 示例运行正常。
- 通过了持续集成 (CI) 测试。

---

## The Story of This Pull Request

这个 PR 的故事始于一个简单的可用性（usability）问题。在 Bevy 引擎的 `FreeCamera` 控制器中，用于捕获鼠标以控制镜头旋转的按钮是鼠标左键（`MouseButton::Left`）。虽然这在功能上有效，但开发者 **Breakdown-Dog** 识别到了一个潜在的长期问题：输入冲突。

问题的核心在于交互模式的约定。在许多 3D 内容创作工具（如 Blender）和游戏引擎（如 Unity、Unreal）中，鼠标左键的默认角色是选择和点击。它用于选取场景中的物体、与 UI 元素交互。相反，鼠标右键则通常被保留用于控制视图本身——旋转、平移摄像机。将两种截然不同的操作（选择和视图控制）绑定到同一个物理按键上，会带来直接的操作冲突。例如，当用户试图点击一个物体时，相机可能会意外旋转；反之，当用户试图旋转视图时，系统可能会误认为他想要选择某个物体。

这个冲突在当时可能没有立即显现，因为 Bevy 的官方示例中可能还没有复杂的场景选择工具。然而，对于一个旨在构建复杂应用和游戏的引擎来说，从一开始就建立清晰、符合直觉的输入映射至关重要。这是一种前瞻性的设计，旨在避免未来的功能迭代引入新的交互时，与现有核心控制器产生不可调和的冲突。

**解决方案** 非常简单且直接：更改配置。开发者没有引入复杂的配置系统或条件逻辑，而是选择修改 `FreeCamera` 结构体 `Default` 实现中的一个常量。这个结构体有一个名为 `mouse_key_cursor_grab` 的字段，它定义了哪个鼠标按键会触发光标的捕获和解锁。PR 所做的就是将这个字段的默认值从 `MouseButton::Left` 改为 `MouseButton::Right`。

这是一种“约定优于配置”（convention over configuration）思想的实践。它为所有新创建的 `FreeCamera` 实例提供了一个符合行业最佳实践和用户心智模型的默认行为。如果某个特定项目需要不同的行为（例如，一个飞行模拟器可能希望用左键开火，中键控制视角），开发者仍然可以显式地设置 `mouse_key_cursor_grab` 字段来覆盖这个默认值。因此，这个改动既提升了默认体验，又保持了完全的灵活性。

从技术实现来看，改动极小，只涉及两行代码：一行在库的默认实现中，另一行在对应的示例文档中。这种同步更新非常重要。示例代码（examples）是用户学习和理解 API 的第一手资料。如果示例中的文档注释（doc comment）与实际代码行为不一致，会给用户造成困惑。PR 作者同时更新了两处，确保了文档与实际行为的一致性，体现了良好的工程习惯。

这个改动的**影响**是立竿见影且积极的。所有使用 `FreeCamera` 且未显式设置 `mouse_key_cursor_grab` 的现有项目，在升级引擎后，其交互逻辑会自动对齐到更标准的模式。这减少了用户需要手动调整配置的负担，并为他们未来的功能开发（如添加物体选择）扫清了一个潜在的障碍。从更深层次看，这个 PR 强化了 Bevy 对用户体验和符合直觉设计的关注。它表明，即使是看似微小的默认值设置，也对构建一个专业、易用的工具生态系统至关重要。
