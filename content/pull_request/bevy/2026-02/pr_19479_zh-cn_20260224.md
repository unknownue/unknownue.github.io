## The Story of This Pull Request

这个Pull Request源于一个明确的需求：为`PlaneMeshBuilder`提供在X和Z轴上独立控制网格细分(subdivisions)的能力。在原始的`PlaneMeshBuilder`实现中，只有一个统一的`subdivisions`字段。这个字段同时控制X轴和Z轴上的细分次数，这意味着生成的网格在每个方向上的分割数量必须相同。对于很多应用场景来说，这限制了灵活性。例如，开发者可能需要创建一个在X方向更精细、在Z方向更粗糙的网格，以优化性能或匹配特定的几何形状。

问题的核心在`PlaneMeshBuilder`的结构体定义中。原结构体只有一个`subdivisions: u32`字段。当调用`build()`方法时，它会基于这个单一的值计算顶点和索引数量：`let z_vertex_count = self.subdivisions + 2;` 和 `let x_vertex_count = self.subdivisions + 2;`。这种对称性强制了网格必须是正方形的分割布局。

解决方案直截了当，正如PR作者和评论者所指出的：将单一的`subdivisions`字段拆分为两个独立的字段：`subdivisions_x`和`subdivisions_z`。这个改动在概念上简单，但需要系统地更新整个API以保持一致性。这包括：
1.  修改结构体定义，用两个新字段替换旧字段。
2.  更新所有构造函数（如`new`, `from_normal_and_size`, `from_size`），将新字段初始化为0。
3.  修改`build()`方法中的网格生成逻辑，使其分别使用`subdivisions_x`和`subdivisions_z`来计算X和Z方向的顶点数。
4.  为向后兼容性，保留原来的`.subdivisions(u32)`方法，但其行为需要更新：同时设置`subdivisions_x`和`subdivisions_z`为相同的值。
5.  新增`.subdivisions_x(u32)`和`.subdivisions_z(u32)`方法，允许独立设置。
6.  为`Meshable` trait的默认实现提供新字段的默认值（0）。
7.  更新所有相关的文档注释，清晰地解释每个新字段的行为。

实现中的关键细节在于`build()`方法。拆分字段后，顶点网格的维度计算变成了：
```rust
let z_vertex_count = self.subdivisions_z + 2;
let x_vertex_count = self.subdivisions_x + 2;
```
这个改动使得后续的顶点位置计算和索引生成能够自然地处理非正方形的网格划分。顶点位置的计算循环 `for z in 0..z_vertex_count` 和 `for x in 0..x_vertex_count` 已经基于独立的计数运行，因此无需其他修改就能适配新的不对称细分。

一个重要的工程决策是如何处理迁移。这个PR引入了一个破坏性变更（breaking change），因为公开的字段名改变了。为了减轻升级负担，作者同时提供了一个迁移指南（migration guide）。指南清楚地说明了变更内容，并给出了代码示例，指导用户如何将直接对`subdivisions`字段的赋值，改为调用`.subdivisions()`方法（该方法保持了旧有的对称细分行为）。这是一个很好的实践，平衡了API改进和开发者体验。

从技术角度看，这个PR展示了如何通过解耦参数来增强API的灵活度，同时保持核心算法不变。网格生成的算法（计算顶点位置、法线、UV和索引）本身不关心细分值是否对称；它只依赖`x_vertex_count`和`z_vertex_count`这两个最终的数字。因此，主要改动都集中在API层和配置层，核心的网格生成逻辑几乎不受影响。这种设计使得功能扩展非常清晰且风险较低。

最终，这个改动为Bevy用户提供了更大的控制权，可以创建更符合其需求的平面网格，同时通过保留便捷的`.subdivisions()`方法和提供迁移指南，确保了良好的向后兼容性和升级路径。它是一个典型的、目标明确的API增强案例。

## Key Files Changed

### `crates/bevy_mesh/src/primitives/dim3/plane.rs` (+57/-11)
这是实现改动的核心文件。主要变更包括将单一的`subdivisions`字段拆分为`subdivisions_x`和`subdivisions_z`，并更新所有相关的方法和文档。

1.  **结构体字段拆分**：这是最基础的变更，为独立控制提供了数据基础。
    ```rust
    // Before:
    pub struct PlaneMeshBuilder {
        pub plane: Plane3d,
        pub subdivisions: u32,
    }
    // After:
    pub struct PlaneMeshBuilder {
        pub plane: Plane3d,
        pub subdivisions_x: u32,
        pub subdivisions_z: u32,
    }
    ```

2.  **构造函数更新**：所有构造函数都需要初始化两个新字段。
    ```rust
    // 以`new`方法为例的变更：
    // Before:
    Self {
        plane: Plane3d { ... },
        subdivisions: 0,
    }
    // After:
    Self {
        plane: Plane3d { ... },
        subdivisions_x: 0,
        subdivisions_z: 0,
    }
    ```

3.  **`build()`方法更新**：网格生成逻辑现在使用独立的计数。
    ```rust
    // Before:
    let z_vertex_count = self.subdivisions + 2;
    let x_vertex_count = self.subdivisions + 2;
    // After:
    let z_vertex_count = self.subdivisions_z + 2;
    let x_vertex_count = self.subdivisions_x + 2;
    ```

4.  **API方法扩展**：新增了独立控制的方法，并更新了原有方法的行为。
    ```rust
    // 原有的`.subdivisions()`方法现在同时设置两个轴
    pub fn subdivisions(mut self, subdivisions: u32) -> Self {
        self.subdivisions_x = subdivisions;
        self.subdivisions_z = subdivisions;
        self
    }
    // 新增的独立控制方法
    pub fn subdivisions_x(mut self, subdivisions: u32) -> Self { ... }
    pub fn subdivisions_z(mut self, subdivisions: u32) -> Self { ... }
    ```

### `release-content/migration-guides/plane3d_subdivisions_xz.md` (+15/-0)
这是一个新增的迁移指南文件，用于指导用户如何将他们的代码从旧的API迁移到新的API。它清晰地指出了破坏性变更的位置并给出了示例。

```rust
// 文件内容展示了迁移方法：
// Before:
builder.subdivisions = 4
// After:
builder.subdivisions(4)
```
这个指南帮助用户理解，直接给字段赋值的旧方式不再有效，需要改为调用方法。同时，调用`.subdivisions(4)`会保持之前的行为（X和Z轴都细分为4）。
